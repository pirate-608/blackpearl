# blackpearl-agent 开题报告

> 选题一：基于 ReAct 范式的多厂商 AI Agent 框架

---

## 一、选题背景与意义

大语言模型（LLM）在自然语言理解和生成方面取得了突破性进展，但纯文本模型存在两大瓶颈：(1) 无法获取实时外部信息，知识受限于训练截止日期；(2) 无法执行精确的数学计算或文件操作。AI Agent 范式通过赋予 LLM "调用工具"的能力，使其从被动回答者转变为主动执行者，成为当前 AI 应用开发的主流方向。

本项目构建了一个完整的教学型 AI Agent 框架 **blackpearl-agent**，核心特点包括：(1) 不依赖任何高层 Agent 框架（如 LangChain），所有核心逻辑手工实现；(2) 同时支持 OpenAI、Gemini、Claude、DeepSeek、Ollama 五类模型后端；(3) 提供 TUI 终端界面和 Web 界面双入口；(4) 实现了多 Agent 协作模式（规划 Agent + 执行 Agent）。

---

## 二、技术背景调研与文献分析

### 2.1 技术现状

#### 2.1.1 ReAct 范式与 Function Calling 的演进

ReAct（Reasoning + Acting）范式由 Yao et al. (2023) 提出，核心思想是让 LLM 交替进行"思考"和"行动"：模型先输出一段推理文本（Thought），然后生成一个动作指令（Action），环境执行后返回观察结果（Observation），模型据此继续推理。这一范式将推理链与工具调用交织在一起，显著提升了复杂多步任务的完成率。

2023 年底，OpenAI 在 GPT-4 中引入了原生 **Function Calling** 能力，与 ReAct 的关键区别在于：

| 维度 | ReAct 范式 | OpenAI Function Calling |
| --- | --- | --- |
| 工具描述方式 | 文本描述嵌入 Prompt | 结构化 JSON Schema |
| 调用触发 | 模型自由决定何时调用 | API 层识别 `tool_calls` |
| 参数规范 | 文本解析，易出错 | JSON 结构化参数，类型安全 |
| 结果回传 | 拼接文本上下文 | `function_call_output` 结构化回传 |

本项目同时实现了三种 API 模式的调用循环：
- **OpenAI Responses API**：使用 `previous_response_id` 串联对话轮次，工具以 `function_call` / `function_call_output` 传递
- **Chat Completions API**：以 `messages` 数组 + `tool_calls` 实现，兼容 Gemini、Ollama
- **Anthropic Messages API**：以 `tool_use` / `tool_result` 协议实现，用于 Claude 和 DeepSeek

#### 2.1.2 多 Agent 协作范式

单 Agent 在处理复杂任务时面临"上下文窗口溢出"和"推理链条过长"的问题。近年来多 Agent 协作成为研究热点：

- **AutoGPT / BabyAGI (2023)**：通过任务分解 + 优先级排序实现自主任务执行
- **MetaGPT (2023)**：模拟软件公司组织结构，多个 Agent 扮演不同角色协作
- **CrewAI / AutoGen (2024)**：提供多 Agent 编排框架，支持角色定义和对话流程

本项目实现的**双层 Agent 架构**属于轻量级多 Agent 协作：一个 Planning Agent 将用户请求分解为 JSON 步骤数组，一个 Execution Agent 逐步执行并传递上下文，最后汇总生成最终回答。这种设计的优势在于：(1) 实现简单，无需复杂的 Agent 间通信协议；(2) 每步独立执行，上下文可控；(3) 执行过程可观测、可调试。

#### 2.1.3 厂商兼容性挑战

不同 LLM 厂商对工具调用的支持程度差异显著：

- **OpenAI**：原生支持 Responses API 和 Chat Completions 两种模式
- **Gemini**：提供 OpenAI-compatible 端点，实际仅支持 Chat Completions 模式
- **Claude**：使用独立的 Messages API，`tool_use` 字段结构与 OpenAI 不同
- **DeepSeek**：在 thinking 模式下 `reasoning_content` 回传存在兼容问题，官方推荐使用 Anthropic-compatible 端点

本项目通过 **Provider Profile + Runner Factory** 模式实现多厂商适配：每个 Provider 定义其 `kind`（`openai_compatible` 或 `anthropic`）和默认配置，Runner Factory 根据 Provider 类型分发到对应的 Runner 实现。

### 2.2 核心原理预研

#### 2.2.1 Prompt 工程与输出约束

Agent 的行为主要由 **System Prompt** 控制。本项目中的 System Prompt 核心约束包括：

```
SYSTEM_PROMPT:
- Use tools when the task requires calculation, factual lookup, or local file access.
- Never invent tool results. If a tool is needed, call it.
- Keep tool arguments valid according to the provided schemas.
- After tool use, explain the result briefly and cite the key tool observation.
```

这些约束实现了三个关键目标：(1) **工具决策**——模型通过语义理解判断何时需要调用工具，而非硬编码规则；(2) **格式约束**——通过 Zod Schema 将工具参数转换为 JSON Schema 7 传给模型，确保参数类型安全；(3) **行为边界**——通过 "never invent" 规则防止幻觉。

在多 Agent 模式下，Planning Agent 的 Prompt 进一步约束输出格式：

```
PLANNER_PROMPT:
- Output ONLY a valid JSON array of step description strings. No other text.
- Each step must be a clear, self-contained instruction.
- Do NOT call any tools. You only produce the plan.
```

此处的关键是 **Prompt-level function calling gate**：虽然 Planning Agent 使用同一个 Runner 实例，但通过 `RunOptions` 传入 `tools: []` 空数组，从 API 层面禁用了工具调用。这保证规划阶段模型专注于任务分解而非执行。

#### 2.2.2 Agent 工具调用决策闭环

单 Agent 的决策闭环流程如下：

```text
┌──────────────────────────────────────────────────┐
│  用户输入                                          │
│    ↓                                              │
│  [Orchestrator] 构建上下文                         │
│    ├─ 短期记忆（最近 N 轮对话）                     │
│    ├─ 长期记忆（关键词检索）                        │
│    └─ 用户请求                                     │
│    ↓                                              │
│  [Runner] 发送请求到模型 API ──→ 流式接收响应       │
│    ↓                                              │
│  模型返回判断：                                    │
│    ├─ 普通文本？ ──→ 作为最终回答输出               │
│    └─ tool_calls？ ──→ 本地执行工具                 │
│                          ↓                        │
│                    [ToolRegistry] Zod 校验参数      │
│                          ↓                        │
│                    [Tool] execute()                │
│                          ↓                        │
│                    工具结果回传模型 ──→ 继续循环     │
│                                                    │
│  (达到 maxSteps 上限 → 抛出 AgentError)             │
└──────────────────────────────────────────────────┘
```

循环设计的关键决策：

1. **串行工具调用** (`parallel_tool_calls: false`)：虽然所有 API 都支持并行工具调用，但串行执行便于教学展示——模型每次只有一个工具调用，步骤清晰可追踪。

2. **失败回传**：工具执行失败时不中断循环，而是将错误信息以标准格式（`{ error: message }`）回传给模型。这使得模型可以尝试修正参数或采用替代方案，实现了基础的 self-correction 能力。

3. **步骤限制**：通过 `AGENT_MAX_STEPS`（默认 6）防止无限循环。这在数学上等价于一个深度受限的搜索树——每个节点是模型的一次决策，分支是选择哪个工具、传什么参数。

#### 2.2.3 多 Agent 协作的决策流程

多 Agent 模式（`/plan` 命令）引入了一个三阶段流水线：

```text
Phase 1: Plan（规划 Agent）
  用户请求 → PLANNER_PROMPT + 无工具
    → 输出: JSON 步骤数组
    → 事件: plan_created

Phase 2: Execute（执行 Agent）
  for each step:
    当前步骤 + 上一步结果（上下文传递）
      → EXECUTOR_PROMPT + 完整工具
      → 事件: step_started / step_completed
      → 结果加入 stepResults[]

Phase 3: Summarize（汇总 Agent）
  原始请求 + 所有步骤结果
    → 无工具，纯文本汇总
    → 事件: assistant_message（最终回答）
```

这种设计的优势在于：
- **关注点分离**：规划、执行、汇总各自独立
- **上下文控制**：每步执行保持上下文精炼，避免超长上下文
- **可观测性**：每个阶段产生独立事件，UI 层可实时展示进度

#### 2.2.4 事件驱动架构与流式输出

项目采用统一的事件总线（EventBus）架构，所有 Runner 将模型响应转换为标准 `AgentEvent`：

| 事件类型 | 触发时机 | UI 效果 |
| --- | --- | --- |
| `assistant_delta` | 每次收到 token | TUI/Web 流式追加文本 |
| `assistant_message` | 模型输出完成 | 最终化回答内容 |
| `tool_call_started` | 工具调用开始 | 活动区显示工具名和参数 |
| `tool_call_finished` | 工具调用完成 | 活动区显示耗时和结果 |
| `tool_call_failed` | 工具调用失败 | 活动区显示错误信息 |
| `plan_created` | 规划完成 | 显示步骤列表 |
| `step_started/completed` | 执行步骤 | 显示执行进度 |

TUI（Ink + React）和 Web（SSE + Vanilla JS）共享同一套事件流，保证了双界面能力完全一致。

#### 2.2.5 记忆系统的设计

项目实现了轻量级双层记忆，适合教学展示：

- **短期记忆**：进程内 `messages[]` 数组，每次注入最近 8 条消息。本质上是滑动窗口上下文。
- **长期记忆**：`.blackpearl/memory.jsonl` JSONL 文件，每轮对话追加一条摘要记录。检索时使用关键词匹配计算相关度分数：

```text
score = Σ(keyword_match × 3 + summary_contains × 1)
```

取 Top-5 结果注入上下文。这不是语义检索（无 embedding），但足够展示"写入 → 召回 → 注入"的记忆闭环。
